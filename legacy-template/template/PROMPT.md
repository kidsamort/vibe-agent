# Промпты для Взаимодействия "Оператор-Исполнитель"

_**Цель этого документа:** Стандартизировать и формализовать команды (промпты) от **Оператора** (человека) к **Исполнителю** (ИИ) для работы в рамках этой проектной документации. Каждый промпт — это четкий приказ к действию, основанный на нашем **[Рабочем Процессе](./template/04-process/01-workflow.md)**._

---

## 🎭 Роли

-   **Оператор:** Вы. Вы принимаете стратегические решения, ставите задачи, проводите ревью и даете финальное одобрение.
-   **Исполнитель:** Я, нейросеть. Я выполняю техническую работу: пишу код, заполняю документы по шаблонам, вношу правки.

---

## 🚀 1. Этап 1: Инициация и Проектирование Задачи

### 1.1. Начало работы над новой задачей

**Назначение:** Запустить работу над новой задачей, создав все необходимые артефакты для проектирования.

```
Исполнитель, приступай к работе над задачей `[TASK-ID: например, DEV-105]`.

**Цель:** `[Краткое описание цели задачи, например: Реализовать экран аутентификации по email]`.
**Спецификация:** Задача реализует требования из документа `[путь/к/спецификации.md]`.
**Эпик:** `[ID эпика из 00-epics-list.md, например, epic-module-a]`.

Начни с **Этапа 1: Подготовка и Проектирование**.
1. Создай папку задачи в `template/05-tracking/04-tasks-pool/` по шаблону `template/05-tracking/03-task-template/`.
2. Заполни паспорт задачи `01-index.md`, указав все метаданные.
3. Подготовь детальное **Проектное Задание** в файле `02-design-spec.md`. Опиши в нем все планируемые изменения: новые компоненты, изменения в состоянии, структура API и т.д.

Ожидаю от тебя готовый `02-design-spec.md` для согласования.
```

### 1.2. Согласование Проектного Задания (Design Spec)

**Назначение:** Подтвердить, что план реализации корректен, и дать разрешение на начало кодирования.

```
Исполнитель, **согласовываю** `02-design-spec.md` для задачи `[TASK-ID]`.

План реализации утвержден. Приступай к **Этапу 2: Реализация**.
1. Напиши код в соответствии с утвержденным планом.
2. Напиши тесты согласно `template/04-process/04-testing-strategy.md`.
3. Веди краткий, но информативный рабочий журнал в `04-implementation-notes.md`.

Ожидаю от тебя готовый код и связанные артефакты для код-ревью.
```

### 1.3. Запрос на доработку Проектного Задания

**Назначение:** Вернуть план на доработку с конкретными комментариями.

```
Исполнитель, **требуются доработки** в `02-design-spec.md` для задачи `[TASK-ID]`.

Внеси следующие правки:
- [Правка 1: например, "Изменить предложенную структуру состояния, чтобы избежать избыточности..."]
- [Правка 2: например, "Добавить обработку ошибки X, которая не была учтена..."]
- [Правка 3: ...]

После внесения изменений предоставь обновленный `02-design-spec.md` для повторного согласования.
```

---

## 💻 2. Этап 2: Реализация и Код-Ревью

### 2.1. Согласование Кода (Code Review)

**Назначение:** Подтвердить, что код написан качественно, и дать разрешение на финальную документацию.

```
Исполнитель, **код-ревью** для задачи `[TASK-ID]` **пройдено успешно**.

Код соответствует стандартам. Приступай к **Этапу 3: Финальная Документация и Завершение**.
1. Заполни `05-final-documentation.md`, подробно описав итоговую реализацию.
2. Убедись, что все сопутствующие документы (диаграммы, журнал) актуальны.

Ожидаю от тебя финальный пакет документов для закрытия задачи.
```

### 2.2. Запрос на доработку Кода

**Назначение:** Вернуть код на доработку после ревью.

```
Исполнитель, по результатам код-ревью задачи `[TASK-ID]` **требуются доработки**.

Внеси следующие исправления в код:
- [Файл 1: `path/to/file.ts`]
  - [Описание правки 1, например, "Функция Y слишком сложная, декомпозируй ее согласно `03-refactoring-guidelines.md`."]
- [Файл 2: `path/to/component.tsx`]
  - [Описание правки 2, например, "Добавь `aria-label` для доступности."]
- [...]

Обнови рабочий журнал `04-implementation-notes.md` и предоставь код на повторное ревью.
```

---

## ✅ 3. Этап 3: Завершение Задачи

### 3.1. Финальное утверждение и закрытие задачи

**Назначение:** Подтвердить выполнение всех этапов и заархивировать задачу.

```
Исполнитель, **финальная документация** для `[TASK-ID]` **утверждена**. Задача полностью завершена.

1. Обнови статус задачи в `01-index.md` на `status: "done"`.
2. Перемести ссылку на задачу на соответствующей канбан-доске из колонки `in-review.md` или `in-progress.md` в колонку `done.md`.
```

## 🛠️ 4. Корректирующие и Общие Промпты

### 4.1. В ответе отсутствуют изменения файлов

**Назначение:** Используется, когда Исполнитель дал текстовый ответ, но не предоставил изменения в коде.

```
Исполнитель, в твоем предыдущем ответе отсутствуют запрошенные изменения в файлах. Пожалуйста, предоставь их в формате XML `<changes>`.
```

### 4.2. Анализ архитектурных изменений после задачи

**Назначение:** Проверить, не повлияла ли выполненная задача на общую архитектуру и не требует ли это обновления документации.

```
Исполнитель, после завершения задачи `[TASK-ID]`, проанализируй итоговую реализацию.

Если были внесены существенные архитектурные изменения (новые модули, изменение потоков данных, принципиально новые решения), которые не отражены в `template/02-architecture/`, предложи обновления для `01-system-overview.md`, `03-data-models.md` или создай новый ADR в `04-decisions/`.

Если таких изменений нет, просто подтверди это.
```

## ♻️ 5. Рефакторинг и Технический Долг

### 5.1. Целевой рефакторинг файла/модуля

**Назначение:** Дать команду на улучшение конкретного участка кода.

```
Исполнитель, проведи рефакторинг файла `[путь/к/файлу.ts]`.

**Обоснование:** `[Причина рефакторинга, например, "Файл превышает 120 строк, нарушен SRP."]`.
**Инструкции:**
1. Руководствуйся принципами из `template/04-process/03-refactoring-guidelines.md`.
2. Обрати особое внимание на "кодовый запах" `[название запаха, например, "Толстый Слайс"]`.
3. Примени паттерн `[название паттерна, например, "Тонкий Слайс, Толстый Сервис"]`.

Предоставь обновленные файлы. Существующая функциональность и тесты не должны быть сломаны.
```

### 5.2. Инициация задачи по техническому долгу

**Назначение:** Начать плановую работу по устранению техдолга.

```
Исполнитель, инициируй новую задачу по техническому долгу `[DEBT-ID]`.

**Эпик:** `[ID эпика из debt-epics-list.md]`.
**Цель:** `[Описание цели, например, "Обновить все зависимости до мажорных версий и исправить совместимость"]`.

Начни с **Этапа 1** и подготовь `02-design-spec.md` с планом работ: какие пакеты будут обновлены, какие риски существуют, и как будет проводиться тестирование.
```
