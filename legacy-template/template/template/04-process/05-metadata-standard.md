# Стандарт Метаданных (YAML Front Matter)

_**Цель этого документа:** Установить единый стандарт для метаданных, которые должны присутствовать в начале каждого Markdown-файла в проекте. Эти данные, известные как "YAML Front Matter", превращают наши документы из простого текста в структурированные, машиночитаемые сущности, формируя **граф знаний** проекта._

---

## 1. Общий Синтаксис

Все метаданные заключаются между двумя строками `---` в самом начале файла. Формат — YAML.

```yaml
---
key1: "value1"
key2: 123
list:
  - item1
  - item2
---

# Заголовок документа...
... остальной контент
```

---

## 2. Базовые (Обязательные) Метаданные

Эти поля должны присутствовать в **каждом** `.md` файле без исключений.

| Поле    | Тип      | Описание                                                                                             |
|---------|----------|------------------------------------------------------------------------------------------------------|
| `id`    | `string` | Уникальный, короткий и человекочитаемый идентификатор документа в kebab-case. **Никогда не меняется.** |
| `title` | `string` | Полное, читаемое название документа, которое будет использоваться для отображения в UI.                 |

**Пример:**
```yaml
---
id: concept-vision
title: "Концепция Проекта"
---
```

---

## 3. Контекстно-зависимые Шаблоны

В зависимости от типа документа, к базовым метаданным добавляются специфичные поля.

### 3.1. Для Доски (Колонки)

_Каждый файл, представляющий колонку на канбан-доске, должен содержать следующие метаданные для своей настройки._

```yaml
---
id: dev-board-in-progress
title: "🚧 В Работе (In Progress)"
icon: "🚧"
color: "#4a90e2" # Синий
wip_limit: 5      # Лимит незавершенной работы (0 - без ограничений)
---
```

### 3.2. Для Справочника Эпиков (`00-epics-list.md`)

_Справочник эпиков является "единым источником правды". Сами данные об эпиках хранятся **в теле файла** в виде YAML-блока для удобства парсинга._

**Метаданные (Front Matter):**
```yaml
---
id: dev-epics-list
title: "Справочник Эпиков: Разработка (DEV)"
---
```

**Тело Файла:**
````markdown
# Справочник Эпиков: Разработка (DEV)

```yaml
- id: "epic-module-a"
  name: "Эпик: Основной Модуль А"
  color: "#4a90e2"
  owner: "tech-lead-id"

- id: "epic-module-b"
  name: "Эпик: Вспомогательный Модуль B"
  color: "#f5a623"
  owner: "backend-lead-id"
```
````

### 3.3. Для Задачи Канбан (`01-index.md` в папке задачи)

_Эти поля превращают текстовый файл в полноценную карточку задачи, похожую на Notion или Jira._

```yaml
---
id: 'DEV-001'
title: "DEV-001: Реализовать основной компонент Модуля А"
epic: 'epic-module-a'   # ОБЯЗАТЕЛЬНО: ID родительского эпика
status: "in-progress"   # inbox | in-progress | done | archive
priority: "high"        # low | medium | high | critical
assignee: "ai-developer"  # ID исполнителя
due_date: "2024-12-31"  # Срок выполнения в формате ГГГГ-ММ-ДД
tags:                   # Список тегов для фильтрации
  - "module-a"
  - "ui"
links:
  - type: 'implements'
    target: 'spec-feature-a'
    description: 'Эта задача реализует UI-часть спецификации "Функционал А".'
---
```

### 3.4. Для Диаграммы (`.md` файл с Mermaid)

```yaml
---
id: diagram-auth-flow
title: "Диаграмма: Поток Аутентификации"
diagram_type: "sequence" # Тип диаграммы: sequence, class, er-diagram, etc.
links:
  - type: 'visualizes'
    target: 'feature-authentication'
    description: 'Эта диаграмма визуализирует процесс, описанный в документе о фиче.'
---
```

### 3.5. Для Архитектурного Решения (ADR)

```yaml
---
id: adr-001-zustand-choice
title: "ADR 001: Выбор Zustand для управления состоянием"
status: "accepted" # proposed | accepted | rejected | deprecated
date: "2024-05-15"
---
```

### 3.6. Для Обычного Документа (Концепция, Гайд)

_К базовым полям можно добавлять поле `owner` для указания ответственного за актуальность документа._

```yaml
---
id: concept-vision
title: "Концепция Проекта"
owner: "project-manager"
---
```

## 4. Система Связей (`links`)

Поле `links` является ключевым для построения графа знаний. Каждая связь — это объект с тремя полями:

- **`type`**: Тип связи. Определяет семантику отношения.
- **`target`**: `id` документа, с которым устанавливается связь.
- **`description`**: (Опционально) Краткое описание, поясняющее контекст связи.

### Стандартные Типы Связей:

| Тип                | Значение                                             |
|--------------------|------------------------------------------------------|
| `based_on`         | "Основано на..." (например, архитектура на концепции) |
| `implements`       | "Реализует..." (задача реализует фичу)               |
| `visualizes`       | "Визуализирует..." (диаграмма визуализирует процесс) |
| `clarifies`        | "Уточняет..." (ADR уточняет часть архитектуры)        |
| `related_to`       | "Связано с..." (общая, неспецифическая связь)        |
| `depends_on`       | "Зависит от..." (одна задача зависит от другой)       |
| `blocked_by`       | "Блокируется..." (одна задача блокируется другой)    |
