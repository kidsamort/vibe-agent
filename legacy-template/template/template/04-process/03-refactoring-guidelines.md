# Руководство по Рефакторингу: Поддержание Чистоты Кода

_**Цель этого документа:** Определить нашу философию и конкретные, действенные правила по улучшению существующей кодовой базы. Рефакторинг — это не отдельная задача, а непрерывный процесс._

---

## 1. Философия: "Правило Бойскаута"

> **Оставляй место стоянки (кодовую базу) чище, чем ты его нашел.**

Если вы работаете с файлом и видите небольшой недочет (плохое имя переменной, сложную для понимания функцию), не проходите мимо. Уделите несколько минут, чтобы это исправить. Постоянные, небольшие улучшения предотвращают накопление технического долга и сохраняют кодовую базу здоровой.

---

## 2. Основные "Кодовые Запахи" (Code Smells) и Как их Устранять

"Кодовый запах" — это признак потенциальной проблемы в коде. Ниже перечислены самые распространенные "запахи" в нашем проекте и способы их устранения.

### 2.1. "Раздутый Файл" (The Bloater)

-   **Запах:** Файл (`.ts` или `.tsx`) становится слишком большим.
-   **Правило: "Правило 120 Строк"**
    > Любой файл, который приближается к размеру в **120 строк кода** (или превышает его), является основным кандидатом на рефакторинг.
-   **Почему это плохо?** Нарушается Принцип Единственной Ответственности (SRP), файл трудно читать и тестировать.
-   **Решение:** Декомпозиция. Разбейте файл на более мелкие, сфокусированные модули.

---

### 2.2. "Толстый Слайс" (The Fat Slice)

-   **Запах:** Файл состояния Zustand (`*.slice.ts`) содержит сложную бизнес-логику внутри своих действий.
-   **Правило: Архитектурный Паттерн "Тонкий Слайс, Толстый Сервис"**
    -   **"Тонкий Слайс" (`*.slice.ts`):** Должен содержать **только** определение данных и действия, которые **делегируют** вызовы чистым функциям из "сервисного" слоя. Он не должен содержать `if`, `for`, сложных вычислений.
    -   **"Толстый Сервис" (`*.service.ts`):** Модуль с чистыми, изолированными бизнес-функциями. Эти функции принимают состояние и другие аргументы, а возвращают **новое, измененное состояние**. Они ничего не знают о Zustand.

-   **Почему это плохо?** Смешивает управление состоянием с бизнес-логикой, делает логику непереиспользуемой и труднотестируемой.
-   **Решение:** Вынесите всю сложную логику из слайса в отдельные, чистые функции в `src/services/`.

**Пример:**

**❌ НЕПРАВИЛЬНО (Толстый слайс):**
```typescript
// user.slice.ts
updateUser: (data) => {
  set(state => {
    // ... ~15 строк сложной логики прямо здесь ...
    const newProfile = { ...state.user.profile, ...data };
    const isValid = validateProfile(newProfile);
    // ... и т.д.
    return { user: { ...state.user, profile: newProfile } };
  });
},
```

**✅ ПРАВИЛЬНО (Тонкий слайс, толстый сервис):**
```typescript
// user.service.ts
export function handleUpdateUser(user, data) {
  // ... ~15 строк чистой, тестируемой логики здесь ...
  return newUserState;
}

// user.slice.ts
import { handleUpdateUser } from '@/src/services/user.service';

updateUser: (data) => {
  set(state => ({
    user: handleUpdateUser(state.user, data)
  }));
},
```

### 2.3. "Магические Числа" и "Строковые Литералы"

-   **Запах:** В коде встречаются необъясненные числа или строки.
    -   `if (user.role === 'admin') { ... }`
    -   `const total = price * 1.2;`
-   **Почему это плохо?** Делает код непонятным и хрупким. Изменение такого значения потребует поиска по всей кодовой базе.
-   **Решение:** Выносите все подобные значения в централизованный файл конфигурации (`src/config/app.config.ts`) или в константы в верхней части файла.

---

## 3. Когда Проводить Рефакторинг?

-   **Непрерывно:** Применяйте "Правило Бойскаута" в каждой задаче.
-   **Перед Новой Фичей:** Если вы собираетесь добавить новую функциональность в существующий, но "грязный" модуль, сначала проведите его рефакторинг. Проще строить на чистом фундаменте.
-   **Выделенные Задачи:** Для крупных архитектурных улучшений (как **TASK-801**) создаются отдельные задачи на рефакторинг.
