# Стандарты Качества Кода

_**Цель этого документа:** Определить единый набор правил и соглашений по написанию кода. Соблюдение этих стандартов **обязательно** для всех участников проекта и является основой для создания чистого, читаемого и поддерживаемого продукта._

---

## 1. Язык и Соглашения по Именованию

-   **Язык:** Весь код, комментарии и документация (JSDoc) пишутся на **русском языке**.
-   **Переменные и Функции:** `camelCase` (`myVariable`, `calculateTotal`).
-   **Компоненты React и Классы:** `PascalCase` (`UserProfile`, `ApiService`).
-   **Типы и Интерфейсы:** `PascalCase` (`interface User`, `type ProductId`).
-   **Константы:** `UPPER_SNAKE_CASE` (`MAX_CONNECTIONS`).
-   **Файлы:**
    -   Компоненты: `PascalCase` (`MyComponent.tsx`).
    -   Остальные `.ts` файлы: `kebab-case` (`api.service.ts`).

---

## 2. Соглашения по Импортам

-   **Абсолютные Пути:** **Всегда** используйте абсолютные пути с алиасом `@/`, который указывает на корень проекта. Это повышает читаемость и упрощает рефакторинг.
    -   **✅ Правильно:** `import { User } from '@/src/types';`
    -   **❌ Неправильно:** `import { User } from '../../types';`

---

## 3. Комментарии и Документация

-   **Объясняйте "Почему", а не "Что":** Хороший комментарий объясняет бизнес-логику или причину выбора нетривиального решения, а не пересказывает код.
    -   **✅ Правильно:** `// Используем debounce, чтобы избежать лишних запросов к API при вводе.`
    -   **❌ Неправильно:** `// Увеличиваем счетчик на 1.`
-   **JSDoc:** Все публичные функции, классы, методы и сложные типы **должны** иметь JSDoc-блок с описанием и, при необходимости, параметрами (`@param`) и возвращаемым значением (`@returns`).

---

## 4. Стилизация (CSS-in-JS с [Styling Library])

-   **Инкапсуляция:** Вся стилизация компонента **должна** быть вынесена в отдельный файл `[ComponentName].styled.ts`. Файл `.tsx` должен содержать только логику и разметку.
-   **Синтаксис:** Используйте функциональный синтаксис `styled('div')\`...\`;`.
-   **Никаких инлайновых стилей:** Использование `style={{ ... }}` **запрещено**, за исключением случаев, когда стили являются динамическими (например, позиционирование элемента).

---

## 5. Принципы Проектирования Компонентов ([Frontend Framework])

-   **Принцип Единственной Ответственности (SRP):** Компонент должен делать только одну вещь. Если он становится слишком большим или сложным, его необходимо декомпозировать.
-   **Разделение на "Умные" и "Глупые":**
    -   **"Умные" (Оркестраторы/Экраны):** Находятся в `src/components/screens/`. Их задача — получать данные из хранилищ, выполнять вычисления и передавать готовые данные в "глупые" компоненты через `props`.
    -   **"Глупые" (Презентационные):** Находятся в `src/components/features/`, `common/`, `ui/`. Они не должны напрямую обращаться к глобальным хранилищам. Они просто отображают то, что им передали.
-   **Типизация Пропсов:** Для каждого компонента, принимающего `props`, должен быть создан файл `[ComponentName].props.ts` с соответствующим интерфейсом.

---

## 6. Управление Состоянием ([State Management Library])

-   **Глобальное vs. Локальное:**
    -   Используйте **глобальное состояние ([State Management Library])** только для данных, которые нужны в нескольких несвязанных частях дерева компонентов (например, информация о текущем пользователе).
    -   Используйте **локальное состояние (`useState`)** для данных, которые не выходят за пределы одного компонента (например, состояние открытой/закрытой вкладки).
-   **Разделение Хранилищ:** Не создавайте одно гигантское хранилище. Разделяйте состояние на логические домены: `useAppStore` (для общих данных приложения), `useUserStore` (для данных пользователя), `useSomeFeatureStore` (для UI, специфичного для конкретной фичи).
-   **Архитектурный Паттерн:** Строго следуйте паттерну **"Тонкий Слайс, Толстый Сервис"**, описанному в `03-refactoring-guidelines.md`.

---

## 7. Отказоустойчивость и Обработка Ошибок

-   **Обязательный `try...catch`:** Любой асинхронный код (`async/await`) и обработчики событий (`onClick`, `onSubmit`), которые могут вызвать ошибку, **должны** быть обернуты в `try...catch`. `ErrorBoundary` не может перехватить такие ошибки.
-   **Не доверяйте API:** Всегда предполагайте, что ответ от внешнего API может прийти в неверном формате или не прийти вовсе. Обеспечьте резервную логику (fallback) или корректное отображение ошибки пользователю.
