# Стратегия Тестирования

_**Цель этого документа:** Определить наш подход к тестированию, чтобы обеспечить высокое качество, надежность и предсказуемость продукта. Мы придерживаемся прагматичного подхода, основанного на концепции **"Пирамиды Тестирования".**_

---

## 1. Философия: Правильный Тест для Правильной Задачи

Пирамида тестирования предполагает, что у нас должно быть много быстрых и дешевых unit-тестов, меньше более медленных и сложных интеграционных тестов, и совсем немного самых медленных и хрупких end-to-end тестов.

```
      / \
     / ▲ \  <-- End-to-End (E2E) тесты (мало)
    /-----\
   / Инте- \ <--- Интеграционные тесты (средне)
  / грация \
 /-----------\
/ Unit-тесты \ <----- Unit-тесты (много)
/_____________\
```

---

## 2. Уровни Пирамиды

### 2.1. Уровень 1: Unit-тесты (Основание)

-   **Что тестируем?** Самые маленькие, изолированные части системы.
    -   **Чистые функции** в `src/utils/` и `src/logic/`. Это наш главный приоритет.
    -   **Отдельные компоненты ([Frontend Framework])** ("глупые" компоненты) в изоляции, проверяя, что они корректно рендерятся при получении разных `props`.
-   **Инструменты:** [Testing Framework, e.g., Vitest], [Component Testing Library, e.g., React Testing Library].
-   **Цель:** Проверить, что каждый "кирпичик" работает правильно сам по себе. Тесты должны быть очень быстрыми.
-   **Покрытие:** Стремимся к **высокому** покрытию кода на этом уровне (>80% для логических модулей).

**Пример (для `billing.logic.ts`):**
```typescript
// billing.logic.test.ts
describe('calculateDiscount', () => {
  it('should apply a 10% discount for premium users', () => {
    const user = { type: 'premium' };
    const price = 100;
    const finalPrice = calculateDiscount(price, user);
    expect(finalPrice).toBe(90);
  });
});
```

---

### 2.2. Уровень 2: Интеграционные тесты (Середина)

-   **Что тестируем?** Взаимодействие нескольких модулей.
    -   **"Умный" компонент + Хранилище:** Проверяем, что при клике на кнопку в компоненте корректно вызывается действие в [State Management Library], и UI обновляется.
    -   **Контроллер/Сервис + Состояние:** Проверяем полный цикл действия: вызов метода, который меняет состояние, что, в свою очередь, влияет на другие части состояния.
-   **Инструменты:** [Component Testing Library, e.g., React Testing Library] (с полной отрисовкой), [Testing Framework, e.g., Vitest]/[Testing Framework, e.g., Jest].
-   **Цель:** Убедиться, что наши "кирпичики" правильно работают вместе.
-   **Покрытие:** Среднее. Мы не пытаемся покрыть все возможные комбинации, а фокусируемся на ключевых сценариях взаимодействия.

**Пример (для `UserListScreen`):**
```typescript
// UserList.integration.test.tsx
it('should display user details when a user is clicked', async () => {
  render(<UserListScreen />);
  // ... имитируем клик по пользователю в списке ...
  fireEvent.click(screen.getByText('Иван Иванов'));
  // ... проверяем, что в панели деталей появился его email ...
  expect(await screen.findByText('ivan@example.com')).toBeInTheDocument();
});
```

---

### 2.3. Уровень 3: End-to-End (E2E) тесты (Вершина)

-   **Что тестируем?** Полные пользовательские сценарии в реальном браузере.
-   **Инструменты:** [E2E Testing Framework, e.g., Playwright] или [E2E Testing Framework, e.g., Cypress].
-   **Цель:** Проверить, что ключевые бизнес-пути работают от начала до конца. Эти тесты медленные и хрупкие, поэтому их должно быть мало.
-   **Ключевые Сценарии для Покрытия:**
    1.  **Регистрация:** Пользователь открывает сайт -> заполняет форму регистрации -> нажимает "Зарегистрироваться" -> видит свой дашборд.
    2.  **Создание Сущности (CRUD):** Пользователь входит в систему -> переходит в раздел "Проекты" -> нажимает "Создать проект" -> заполняет форму -> видит новый проект в списке.

---

## 3. Когда Писать Тесты?

Тесты являются неотъемлемой частью рабочего процесса. Они должны быть написаны в рамках той же задачи, что и основной код. Задача не считается выполненной, пока для нее не написаны соответствующие тесты.
