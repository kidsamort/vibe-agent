# 🤖 Julius: Workflow & Agent Coordination

Julius отвечает за организацию параллельной работы нескольких агентов. Чтобы проект не превратился в хаос, следуйте этим правилам.

## 🛠️ Work Stages (Sprints)

Мы разбиваем работу на этапы (Спринты). Зачем это нужно?

1. **Изоляция**: Агенты работают над независимыми частями системы.
2. **Контроль качества**: В конце каждого этапа происходит синхронизация (Sync Point).
3. **Зависимости**: Один этап подготавливает "почву" для следующего. К концу Спринта 1 все агенты Спринта 2 должны иметь рабочее API для работы с конфигом.

## 🚩 Sync Points (Точки синхронизации)

В конце каждого спринта Antigravity (или пользователь) выполняет:

- Проверку всех Pull Request'ов.
- Разрешение конфликтов (особенно в `src/index.ts` и `src/core/config.types.ts`).
- Валидатор API: Проверка, что новый функционал не сломал старый.

## 🤝 Coordination Rules

1. **Branching**: Каждый агент создает ветку `jules/[agent-name]-[feature]`.
2. **Pull Requests**: Никаких прямых пушей в `main`. Только через PR.
3. **Documentation**:
   - Агент ОБЯЗАТЕЛЬНО обновляет `task.md` при завершении своей части задачи.
   - Агент пишет отчет в `README.md` или создает новый артефакт, если задача сложная.
4. **No Side-Effects**: Агенту запрещено менять файлы, не относящиеся к его задаче, если это не согласовано заранее.
