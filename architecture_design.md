# Архитектура Vibecoding CLI (vibe-agent) — v2

## 1. Технологический Стек

- **Runtime**: [Bun](https://bun.sh/)
- **CLI Framework**: [Consola](https://github.com/unjs/consola)
- **Data Formats**:
  - **YAML** (`agent.config.yml`): Основной юзер-фрейндли конфиг.
  - **JSON**: Внутренняя структура данных и метаданные плагинов.
  - **XML-структуры**: Будут использоваться внутри шаблонов для четкого разграничения блоков инструкций для ИИ (prompt formatting).
- **Парсинг документов**: `gray-matter` + расширенный парсер для плагинов.

---

## 2. Плагинная Система (Extensibility)

Чтобы проект мог «нарастать жиром» через Open Source, внедряется двухуровневая система плагинов:

### Локальные плагины (`.agents/plugins/`)

Пользователи могут создавать свои скрипты и шаблоны прямо в проекте. CLI автоматически подхватывает всё, что лежит в этой папке.

### Внешние плагины (NPM)

Поддержка установки сторонних пакетов: `vibe plugin add @user/vibe-plugin-notion`.
Плагин может содержать:

- **Templates**: Наборы документов (MDX/XML).
- **Rules**: Специфичные правила для стека.
- **Actions**: Bun-скрипты для автоматизации (например, синхронизация с API).

---

## 3. Гибридная Шаблонизация (XML + JSON + MDX)

Мы принимаем во внимание качество промпт-инжиниринга:

- **Для Человека**: Итоговый результат в `docs/` остается в **Markdown**.
- **Для ИИ**: Генерация правил и инструкций (`.agentrules`) будет использовать **XML-теги** для структурирования контекста. Это исключает путаницу и помогает агенту четко понимать границы правил.
- **Внутри Шаблонов**: Используем JSON-схемы для описания переменных и ожидаемого поведения, чтобы агент мог оперировать типизированными данными.

## 4. Модульная Структура Проекта (Anti-Monolith)

Чтобы избежать превращения проекта в неподдерживаемый монолит, мы используем паттерн **"Команда-как-Директория"** и выносим бизнес-логику в независимые сервисы.

```text
src/
  commands/             # Точки входа для CLI
    init/
      index.ts          # Регистрация команды
      logic.ts          # Оркестрация процесса
      steps/            # Модульные шаги интерактива
        get-project-name.ts
        select-stack.ts
        configure-tracking.ts
    plugin/
      index.ts
      steps/            # Шаги для плагинов
        select-plugin.ts
        confirm-install.ts
      actions/          # Чистая логика (Side Effects)
        add-plugin.ts
        remove-plugin.ts
    sync/
      ...
  services/             # Переиспользуемая бизнес-логика (Сервисы)
    config.service.ts   # Работа с agent.config.yml
    template.engine.ts  # Рендеринг (Hydration)
    plugin.manager.ts   # Логика плагинов
    skill.runner.ts     # Исполнение скриптов (Skills)
```

---

## 5. Паттерны Проектирования

1.  **Command-Wizard Pattern**: Сложные команды (как `init`) разбиваются на атомарные "Шаги" (Steps). Каждый шаг — это чистый интерфейс: запрос данных -> валидация -> передача в логику.
2.  **Logic-UI Separation**: Файлы в `steps/` отвечают только за Consola (ввод/вывод), файлы в `actions/` или `logic.ts` — за реальные изменения в файловой системе.
3.  **Service Registry**: Вся бизнес-логика инкапсулирована в сервисы, что делает CLI-слой очень тонким и лёгким.
4.  **Composition over Monolith**: Мы собираем команды из маленьких кирпичиков (шагов и сервисов).

## 5. Константы и Глобальный Конфиг

`agent.config.yml` расширяется для поддержки плагинов:

```yaml
project:
  name: "MyProject"

# Глобальные константы для всех плагинов
constants:
  PKG_MANAGER: "bun"
  FORMAT: "xml" # Режим общения с агентом: xml | json
  LANG: "ru"

# Список активных плагинов и их настройки
plugins:
  - name: "base-docs"
  - name: "@vibecoding/kanban"
    config:
      provider: "local"
  - name: "my-custom-plugin" # Локальный плагин
```

## 6. Почему это взлетит (Open Source Ready)

- **Легкий вход**: Чтобы создать плагин, достаточно создать папку с `plugin.json` и набором шаблонов.
- **Версионность**: Константы и конфиг изолируют логику шаблонов от кода CLI. Мы можем обновлять CLI, не ломая пользовательские шаблоны.
- **.agents/workflows/**: Исполнимые Markdown-ворклоу для агента.
- **.agents/skills/**: Библиотека исполняемых скриптов (Bun/TS/Python). ИИ может сам писать новые «навыки» здесь для автоматизации рутины.
- **.agents/hooks/**: Логика валидации документации.

---

## 7. Паттерн "Skills" (Экономия токенов)

Для оптимизации затрат пользователя, фреймворк поддерживает выполнение тяжелых операций через скрипты:

- **Batch Processing**: Вместо того чтобы просить ИИ отредактировать 50 файлов по одному, агент пишет скрипт на Bun, который делает это локально.
- **Data Collection**: Скрипты для сбора статистики проекта или генерации графа связей.
- **Execution Engine**: CLI `vibe` включает команду `vibe run <skill-name>`, которая запускает скрипты из `.agents/skills/` в безопасном окружении.
- **Инструкции в Rules**: Правила `.agentrules` содержат пункт: _"Если действие требует массовых правок или сложного анализа — напиши и выполни скрипт в `.agents/skills/`"_.
